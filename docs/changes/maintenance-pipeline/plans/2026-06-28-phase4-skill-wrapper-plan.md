## Goal

Author the thin `harness-maintenance-pipeline` Tier 2 skill that shells to the already-shipped `harness maintenance run` CLI: it runs `harness maintenance run --json` (overdue, report mode), parses the consolidated report, presents a triaged human-readable summary, asks the human **in plain text** which tasks to fix, then re-invokes `harness maintenance run --only <ids> --fix` — persisting nothing of its own.

## Scope

Phase 4 ONLY. Phases 1–3 are shipped: `TaskRunner` run mode, `selectTasks` + `excludeFromHumanSweep`, and the full `harness maintenance run [taskId...]` subcommand (with `--json` emitting the `ConsolidatedReport` object to stdout and writing `.harness/maintenance/last-run-summary.json`). This phase authors a SKILL.md + skill.yaml on top, registers it as Tier 2, and regenerates the derived platform/plugin/catalog artifacts. NOT in scope: any CLI behavior change, real AI-fix dispatch (a documented repo-wide stub — see Grounding), and Phase 5 docs/knowledge guides.

## Grounding (verified against actual code)

- **The CLI `--json` schema the skill parses is `ConsolidatedReport`** (`packages/cli/src/commands/maintenance-run.ts:270-277`), emitted to **stdout** and also written to `.harness/maintenance/last-run-summary.json`:
  ```ts
  interface ConsolidatedReport {
    generatedAt: string;
    mode: 'report' | 'fix';
    fix: boolean;
    exitCode: 0 | 1;
    tasks: ReportRow[]; // pre-sorted: failures-first, then findings-desc, then id
    overdueNowCurrent: string[]; // ids that were overdue but are now marked current
  }
  interface ReportRow {
    // maintenance-run.ts:260-268
    taskId: string;
    status: 'success' | 'failure' | 'skipped' | 'no-issues'; // RunResult['status'] (types.ts:128)
    findings: number;
    fixed: number;
    prUrl: string | null;
    summary: string; // error text, or `${findings} finding(s)`, or 'clean'
    error?: string;
  }
  ```
- **There is NO per-finding `severity` or `domain` field.** The spec says "grouped by severity/domain," but the emitted schema carries only a per-task `status` + a `findings` **count** + a free-text `summary`. The skill therefore triages by (a) `status` tier (`failure` = could not execute → `findings > 0` = needs attention → `no-issues`/`success`/`skipped` = clean) and (b) `findings` descending, and approximates "domain" from the task id (the registry id encodes the concern: `dead-code`, `doc-drift`, `dependency-health`, `security`, `hotspots`, `project-health`, …). This is the honest mapping; see Concerns.
- **`--fix` dispatch is a documented stub repo-wide.** Phase 3 (`ADR 0050`, handoff D-P3-1) threads `mode: 'fix'` into the same `TaskRunner` but constructs no `PRManager`; the only `AgentDispatcher` in the repo logs "skill dispatch integration pending." `harness maintenance run --only <ids> --fix` prints to **stderr**: `--fix: AI fix-agent dispatch is not yet wired (executor dispatcher is a stub repo-wide); checks ran, no PRs were opened.` (`maintenance-run.ts:365-366`). The skill must still be authored to call `--fix` (the seam is ready) **and surface that warning honestly** rather than claim a fix was applied.
- **Exit codes:** `0` = completed (findings are not failures), `1` = a task failed to execute, `2` = invalid invocation (`maintenance-run.ts` / `ExitCode` in `utils/errors.ts`). The skill reads exit code to distinguish "clean run with findings" from "a check crashed."

## Decisions (this phase)

- **D-P4-1 — Triage taxonomy is `status` tier + `findings` count, not a fictional severity field.** Because `ConsolidatedReport` carries no per-finding severity/domain, the SKILL.md groups rows into three honest buckets — **Failed to execute** (`status === 'failure'`), **Needs attention** (`findings > 0`), **Clean** (everything else) — and within "Needs attention" orders by `findings` descending (the rows already arrive pre-sorted). "Domain" is derived from the task id, not invented.
- **D-P4-2 — The skill reads the report from stdout of `--json`, with `.harness/maintenance/last-run-summary.json` as the documented fallback.** Single parse path; the Read tool is declared so the agent can fall back to the persisted file if stdout capture is awkward on a platform. The skill writes nothing — the CLI owns all artifacts (spec §4).
- **D-P4-3 — `--fix` is offered but its stub status is surfaced verbatim.** After the human selects ids, the skill runs `harness maintenance run --only <ids> --fix`, then reports the run's `fix`-mode result AND echoes the CLI's stderr stub warning, so the human is never told a PR was opened when none was. The skill never claims to fix; it relays what the CLI did.
- **D-P4-4 — `type: flexible`, `stability: draft`.** Per spec §"Skill shape". The triage flow adapts to what the report contains (nothing overdue → stop early; failures present → surface those first), so flexible is correct. `draft` signals the wrapper is new and its `--fix` leg rides a stub.
- **D-P4-5 — `cli:`/`mcp:` blocks included for parity** even though the primary entry is the slash command. `harness skill run harness-maintenance-pipeline` should resolve like every other Tier 2 skill; the blocks cost nothing and keep `harness skill list` / MCP discovery uniform. No `depends_on` (spec: it shells to the CLI, not to an executor skill).

### Task 2: Author `SKILL.md` (thin-wrapper triage flow)

**Depends on:** Task 1 | **Files:** `agents/skills/claude-code/harness-maintenance-pipeline/SKILL.md`

1. Create the file with EXACTLY this content (the human-ask sections are deliberately plain-text; do not introduce any `emit_interaction`/`AskUserQuestion` ask, and never write the literal `type:` "question"/"confirmation"):

   ````markdown
   # Harness Maintenance Pipeline

   > One on-demand entry point for project maintenance. Runs the checks that are actually overdue, triages what needs attention, and asks you — in plain text — which to fix. A thin wrapper over `harness maintenance run`; it owns no state of its own.

   ## When to Use

   - You want to answer "which maintenance did I forget to run?" without standing up an orchestrator.
   - At a milestone or before a release, to sweep overdue health checks (dead code, doc drift, dependency health, security, hotspots, project health).
   - When you want a report first and an explicit, opt-in fix step — never a surprise PR.
   - NOT for running a single known check — call that skill or `harness maintenance run --only <id>` directly.
   - NOT for the autonomous cron fix-and-PR path — that is the orchestrator scheduler's job; this is the human-invoked, report-first path.

   ## What this skill does NOT do

   - It does not implement any maintenance check, scheduler, or registry — the CLI and the existing 22-task registry are the single source of truth.
   - It does not write artifacts. The CLI writes `.harness/maintenance/last-run-summary.json` and records history; this skill only reads and relays.
   - It does not open PRs. Fixes are opt-in via the explicit fix step, and even then dispatch is currently a documented stub (see Phase 3 / FIX).

   ## Process

   ### Phase 1: SWEEP — run the overdue report

   1. Run the report-mode sweep and capture stdout:

      ```bash
      harness maintenance run --json
      ```

      No flags = overdue, sweep-eligible tasks only, report mode (no PRs). Capture both stdout (the JSON report) and the exit code.

   2. Parse stdout as a `ConsolidatedReport`:

      ```ts
      // shape emitted by the CLI (packages/cli/src/commands/maintenance-run.ts)
      {
        generatedAt: string;
        mode: 'report' | 'fix';
        fix: boolean;
        exitCode: 0 | 1;
        tasks: Array<{
          taskId: string;
          status: 'success' | 'failure' | 'skipped' | 'no-issues';
          findings: number;       // a COUNT — there is no per-finding severity
          fixed: number;
          prUrl: string | null;
          summary: string;        // 'clean' | `${n} finding(s)` | error text
          error?: string;
        }>;
        overdueNowCurrent: string[];
      }
      ```

      If capturing stdout is awkward, read the same object from `.harness/maintenance/last-run-summary.json` with the Read tool.

   3. Read the **exit code**: `0` = the sweep completed (findings are NOT failures); `1` = at least one check failed to execute; `2` = invalid invocation (a bug in how this skill called the CLI — fix the invocation, do not report it as a finding).

   4. If `tasks` is empty, tell the human "All maintenance is current — nothing overdue." and stop. Do not invent work.

   ### Phase 2: TRIAGE — present a human-readable summary

   There is no per-finding severity in the report, so triage by **status** then **findings count** (rows arrive pre-sorted failures-first, then findings-descending). Bucket the tasks:

   - **Failed to execute** — `status === 'failure'`. The check itself crashed; surface `summary`/`error`. These are the most urgent: a finding you cannot see is worse than one you can.
   - **Needs attention** — `findings > 0` (and not a failure). Order by `findings` descending. Derive a coarse domain from the task id (e.g. `dead-code`, `doc-drift`, `dependency-health`, `security`, `hotspots`, `project-health`).
   - **Clean** — `status` is `no-issues`/`success`/`skipped` with `findings === 0`. Summarize as a count ("8 checks clean"); do not enumerate unless asked.

   Render a compact summary, for example:

   ```
   Maintenance sweep (report mode) — 18 overdue tasks ran, exit 0

   Failed to execute (1):
     stale-constraints   — check crashed: <summary>

   Needs attention (3):
     dead-code           — 12 finding(s)
     doc-drift           —  4 finding(s)
     dependency-health   —  1 finding(s)

   Clean (14): project-health, hotspots, security, …
   ```

   If `overdueNowCurrent` is non-empty, add a one-line footer: "N tasks were overdue and are now marked current."

   ### Phase 3: FIX — ask in plain text, then opt in

   1. **Ask the human in plain text, in your own reply.** Do NOT use `emit_interaction` and do NOT use `AskUserQuestion` for this — those channels do not reliably reach the human (an `emit_interaction` ask collapses to "Called harness" and the human never sees the question). Write the question as ordinary text, listing the actionable task ids (the "Failed to execute" and "Needs attention" buckets) and asking which, if any, to fix. For example:

      > These tasks have findings: `dead-code` (12), `doc-drift` (4), `dependency-health` (1). Which would you like me to fix? Reply with a comma-separated list of task ids, "all", or "none".

      Then stop and wait for the human's reply. Do not proceed to a fix without an explicit answer.

   2. **If the human picks tasks**, re-invoke the CLI in fix mode, scoped to exactly those ids:

      ```bash
      harness maintenance run --only <comma,separated,ids> --fix
      ```

   3. **Relay the result honestly.** `--fix` currently threads fix mode through the same executor but the AI fix-agent dispatch is a documented repo-wide stub: no PRs are opened. The CLI prints a warning to stderr:

      > `--fix: AI fix-agent dispatch is not yet wired (executor dispatcher is a stub repo-wide); checks ran, no PRs were opened.`

      Surface that warning verbatim. Report what the CLI actually did (`fixed` counts, any `prUrl`) — never claim a fix was applied or a PR opened when the report does not show one.

   4. **If the human picks "none"**, stop. The report stands on its own.

   ## Gates

   - **Plain-text ask only.** The "which to fix?" question must be ordinary text in your reply. Never route it through `emit_interaction` or `AskUserQuestion`.
   - **Report before fix, always.** Never pass `--fix` on the first sweep. Fix is opt-in after the human sees the report and chooses.
   - **No invented findings.** Report only what the `ConsolidatedReport` contains. If `tasks` is empty, say so and stop.
   - **No self-persistence.** Do not write `.harness/maintenance/*` or any report file — the CLI owns artifacts.
   - **Honest fix status.** If the stub warning appears, relay it. Do not imply a PR was opened.

   ## Success Criteria

   - Ran `harness maintenance run --json`, parsed the consolidated report, and read the exit code.
   - Presented a triaged summary bucketed by status (failed / needs-attention / clean) and ordered by findings count.
   - Asked the human in plain text which tasks to fix, and waited for a reply.
   - On opt-in, re-invoked `--only <ids> --fix` and relayed the result plus any stub warning verbatim.
   - Wrote no files of its own.

   ## Rationalizations to Reject

   | Rationalization                                        | Reality                                                                                                       |
   | ------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------- |
   | "I'll just run `--fix` directly to save a round-trip." | Report-first is the whole point. A surprise fix on an on-demand sweep is exactly the hazard this path avoids. |
   | "I'll ask via `emit_interaction` so it's structured."  | That ask never reaches the human — it collapses to "Called harness". Ask in plain text.                       |
   | "The report shows findings, so I'll say I fixed them." | `--fix` dispatch is a stub today; relay what the CLI reports, including the no-PR warning. Do not overclaim.  |
   | "There's no severity field, so I'll assign my own."    | Triage by the status + findings count the report actually provides. Do not fabricate a severity taxonomy.     |
   ````

## Risks & Concerns

- **Schema↔spec gap (severity/domain):** The spec asks for a summary "grouped by severity/domain," but `ConsolidatedReport` has no severity/domain field — only `status`, a `findings` count, and a free-text `summary`. The plan resolves this by triaging on status + findings count and deriving domain from the task id (D-P4-1). - **`--fix` is a stub:** The skill is authored to call `--fix` (the seam is ready) but the underlying dispatcher is a documented repo-wide no-op. The SKILL.md surfaces the stub warning verbatim so the human is never misled. When a real dispatcher lands, no skill change is needed.
